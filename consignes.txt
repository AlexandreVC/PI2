1. Contexte et justification 
La cybersécurité est un enjeu critique. Les équipes doivent traiter un grand volume 
de vulnérabilités issues de scans (Nmap, Nessus, OpenVAS) et bases publiques 
(CVE, NVD). Or, l’analyse manuelle est chronophage. L’objectif est de développer un 
agent intelligent, basé sur l’IA générative, capable d’assister les analystes dans : 
● la compréhension des vulnérabilités (nature, criticité, impact métier), 
● la priorisation selon le contexte, 
● la proposition de mesures de remédiation, 
● la génération automatique de rapports adaptés (exécutif et technique). 
2. Objectifs du projet 
● Concevoir un pipeline automatisé d’analyse de vulnérabilités (import, 
enrichissement, priorisation). 
● Développer un agent IA génératif spécialisé dans l’explication et la 
remédiation. 
● Produire deux rapports distincts : 
○ Rapport exécutif → synthétique, orienté décisionnel (management). 
○ Rapport technique → détaillé, pour les administrateurs sécurité. 
● Tester la robustesse de l’agent (données incomplètes, faux positifs, 
hallucinations IA). 
3. Périmètre et livrables 
3.1 Périmètre inclus 
● Analyse des résultats de scans (Nmap, Nessus, OpenVAS). 
● Enrichissement avec bases publiques (CVE, NVD, CVSS). 
● Intégration d’un modèle de langage (LLM/API open-source ou cloud). 
● Génération de rapports différenciés. 
3.2 Périmètre exclu 
● Tests sur des systèmes en production (uniquement environnements simulés). 
● Déploiement en conditions réelles (prototype uniquement). 
3.3 Livrables attendus 
● Prototype fonctionnel : pipeline + agent IA + génération de rapports. 
● Rapport technique détaillé (architecture, choix technologiques, résultats). 
● Rapport exécutif (vulgarisé, orienté managers). 
● Démonstration finale sur scénario simulé. 
4. Contraintes 
● Respect du cadre pédagogique (pas d’attaques réelles). 
● Outils privilégiés : open-source (limitation des coûts). 
● Respect des délais de chaque sprint (6 semaines). 
● Documentation claire et traçabilité des choix techniques.

2.3. Architecture & Outillage Cyber (Pôle Cyber - Sacha & Romain) 
En parallèle du développement IA, le pôle Cyber a structuré l'environnement source 
qui alimente l'agent. 
 Architecture de Test (Laboratoire Isolé) : Conception et déploiement d'un 
réseau virtuel sécurisé comprenant une machine attaquante (Kali Linux) et 
des cibles volontairement vulnérables (Metasploitable 2, Windows Server, 
DVWA) pour générer des logs d'attaques réalistes. 
 Benchmark des Outils de Scan : Nous avons testé plusieurs solutions du 
marché (Nmap, Nessus, OpenVAS, Masscan) pour évaluer la qualité des 
rapports générés. 
 Sélection Finale : 
o Retenus : Nmap (pour la cartographie) et Nessus Essential (pour les 
vulnérabilités) ont été validés pour la structure de leurs exports 
(XML/JSON) qui sont facilement "parsables" par notre IA. 
o Écartés : Des outils comme Masscan ont été écartés après tests, 
jugés trop spécifiques aux très grands réseaux et moins pertinents pour 
notre périmètre d'analyse fine. 
3. Méthodologie de Test & Validation 
(Basé sur les travaux du Pôle IA - Alexandre & Rayane) 
3.1. Protocole Expérimental (Code Vulnerability) 
Nous avons développé un banc de test automatisé utilisant le dataset 
CyberNative/Code_Vulnerability_Security_DPO (paires de code sécurisé/vulnérable). 
 Gestion d'Erreurs : Implémentation d'un mécanisme de "Retry" (Regex 
fallback) pour parser le JSON si le modèle échoue, garantissant 0 erreur 
technique lors de l'exécution. 
 Prompting : Rôle "Cyber Security Expert" avec contrainte de sortie JSON 
stricte. 
3.2. Justification des Hyperparamètres (Réponse au Feedback Tuteur) 
Suite à la demande du tuteur école, voici la justification technique de nos réglages : 
 Température fixée à 0. 
 Référence théorique : Dans l'échantillonnage de tokens (Temperature 
Sampling), une valeur proche de 0 (Argmax) rend le modèle déterministe. Il 
sélectionne systématiquement le token ayant la probabilité la plus élevée. 
 Intérêt Métier : En cybersécurité, la créativité est un risque (hallucination). 
Nous cherchons la reproductibilité (10 analyses du même log doivent donner 
10 fois le même résultat) et la précision factuelle. 
3.3. Définition des Métriques d'Évaluation 
Pour clarifier la lecture des résultats auprès des parties prenantes, voici les 
définitions appliquées : 
 Accuracy (Justesse) : Taux global de bonnes réponses. Le modèle a-t-il 
raison globalement ? 
 Precision (Fiabilité) : Proportion de vraies vulnérabilités parmi les alertes 
levées. Si le modèle crie "Au loup", y a-t-il vraiment un loup ? (Évite les Faux 
Positifs). 
 Recall (Exhaustivité) : Proportion de vulnérabilités réelles détectées. Le 
modèle a-t-il raté des failles ? (Évite les Faux Négatifs).
